## 2. The duplication bug

**Symptom (②, user report).** On long dictation (EN and 中文) the injected text contained
duplicated chunks the user never repeated — the two copies *slightly different* (`in my`→`my`,
`其`→`族`, `tape`→`tapes`), which is the tell: two recognition passes over the same audio, not the
user repeating. `[uat]`

**Root cause (① + interpretation).** `SFSpeechRecognizer` replaces its streaming hypothesis with a
fresh, shorter string after a pause (a silent endpoint reset, no `isFinal`). We commit the prior
segment across that reset so long dictation isn't lost — but the reset is sometimes a **rewind**: the
fresh string re-transcribes audio already in the committed text. The old `committed + " " + segment`
join then duplicated the overlapping chunk. Proven with a live capture:
`cb#606 … BACKWARD-RESET commit→len=2686 OVERLAP=true probe="This kind of activ"`
`[harness:repro_long.log:1216]`.

Two interpretations were **tested and discarded** as the cause:
- the request-restart-at-cap path — the on-device recognizer ran one continuous task for 200 s+ on
  macOS 15.6 with no ~1-min cap, so restart never fired `[harness:repro_long.log]` (③→①, falsified).
- a first fix that anchored on the segment's exact 24-char prefix — real rewinds change words
  *throughout*, including the leading word, so it kept missing in UAT `[uat]`.

**Fix (①).** Extracted the composition logic into a pure, framework-free
`VoiceInputCore.TranscriptComposer` (unit-testable without a mic) `[file:Sources/VoiceInputCore/TranscriptComposer.swift]`;
`SpeechTranscriber` delegates to it under its existing lock `[file:Sources/VoiceInput/Speech/SpeechTranscriber.swift]`.
Overlap suppression finds the **longest identical run** shared by the committed tail and the new
segment and splices there — robust to word-level divergence and changed leading words. Two guards:
the run must be ≥ 24 chars (so intentional short repeats survive), and the segment from the run
onward must be long enough to re-cover the committed tail it replaces (so trailing text is never
dropped) `[file:TranscriptComposer.swift:compose]`.

**Verification.**
- **①** 49 checks pass, including the exact UAT strings (tape/tapes, Cozy/task-test, 其/族) and the
  intentional-repeat / data-loss guards `[test:Tests/VoiceInputCoreTests/main.swift]`.
- **①** No regression on normal speech: a clean single-pass harness run is byte-identical pre- and
  post-fix — `len=1482` both `[harness:repro.log]` `[harness:harness2_single.log]`.
- **②** User re-ran long EN + ZH dictation: chunk duplication gone; intentional `remember to…` /
  `请记得写…` lists preserved `[uat]`.

## 5. Accepted tradeoffs / known limits (③ unless noted)

- **Sub-24-char repeats are left alone** — Apple's own within-hypothesis stutters (`perch perch`,
  `在世界排在世界排名`) stay, deliberately, so intentional short repeats aren't destroyed `[uat]`.
- **Identical short phrase + ~1 s pause keeps one copy** — an identical-length "reset" is
  indistinguishable from a no-op revision (which streams constantly); committing it would duplicate
  normal partials. Continuous speech keeps both. Documented + tested as expected `[test:main.swift]`
  `[uat]`.
- **8.7 injection is PID-based, not focused-element-level** — the audit's element-tracking and
  restore-timing `changeCount` ideas were judged consistent with the current design and not adopted.
